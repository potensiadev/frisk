## 1. Executive Summary

본 문서는 외국인 유학생 리스크 관리 ERP의 비즈니스 요구사항을 정의합니다. 이 시스템은 유학원 내부 운영용 ERP로서, 네팔 유학생들의 체계적 관리와 대학교 국제처에 대한 신뢰 구축을 목표로 합니다.

| 구분 | 내용 |
|------|------|
| **차별화 포인트** | 타 유학원 대비 체계적인 학생 관리 시스템 보유 |
| **핵심 메시지** | 리스크를 최소화하려면 우리 유학원을 선택해야 한다 |
| **수익 모델** | 네팔 유학원으로부터 학생 1인당 수수료 수취 |

## 2. 사업 배경

### 2.1 시장 현황

한국 대학교들은 외국인 유학생 유치에 어려움을 겪고 있으며, 유학원을 통해 입학 지원을 받고 있습니다. 그러나 외국인 유학생들의 불법체류 문제가 대학 명성에 심각한 영향을 미치고 있습니다.

### 2.2 대학교의 Pain Point

- 외국인 유학생 불법체류 시 대학 비자 발급 스폰서로서 명성 훼손
- 우수인증대학/인증대학 선정에 부정적 영향
- 리스크 관리 및 예방 방법 부재
- 유학원 공급 학생에 대한 체계적 모니터링 불가

### 2.3 시장의 현실

90% 이상의 외국인 유학생들이 학업보다 취업이 주 목적입니다. 대학교들도 이를 인지하고 있으며, 합법적인 아르바이트를 통해 학업과 취업을 병행하기를 희망합니다. 그러나 학생들이 아르바이트 허가 절차를 모르거나 무시하여 불법 취업 문제가 발생하고 있습니다.

## 3. 솔루션 개요

### 3.1 가설

외국인 유학생들의 리스크를 지속적으로 관리할 수 있는 시스템이 있다면, 대학교 국제처는 자체적으로 해결하거나 예방할 수 없는 리스크 관리 문제에서 자유로워질 것입니다.

### 3.2 솔루션

외국인 유학생 리스크 관리 ERP를 자체 개발하여, 네팔 유학생들의 체계적 관리, 결석 알림 서비스, 월간 리포트 제공을 통해 대학교 국제처와의 신뢰를 구축합니다.

| 역할 | 주체 | 책임 |
|------|------|------|
| 시스템 관리자 | 유학원 법인 (본인) | 전체 시스템 관리, 대학교 계정 발급 |
| 운영자 | 네팔 유학원 대표 | 학생 정보 입력, 결석 기록, 증빙 업로드 |
| 열람자 | 대학교 국제처 | 소속 학생 정보 조회 (선택적) |

## 4. 타겟 고객

### 4.1 1차 타겟: 대학교 국제처

- 외국인 유학생 학부 프로그램 운영 대학교
- 외국인 유학생 어학연수 프로그램 운영 대학교

### 4.2 주요 이해관계자

| 이해관계자 | 니즈 | 제공 가치 |
|------------|------|-----------|
| 대학교 국제처 | 불법체류 리스크 최소화, 인증대학 유지 | 체계적 관리 리포트, 실시간 결석 알림 |
| 네팔 유학원 | 학생 관리 효율화, 안정적 수수료 수익 | 통합 관리 시스템, 업무 체계화 |
| 네팔 유학생 | 비자 유지, 합법적 아르바이트 | 아르바이트 허가 서류 안내 및 제공 |

## 5. 핵심 기능 요구사항

### 5.1 사용자 및 권한 관리

| 사용자 유형 | 인증 방식 | 권한 | 비고 |
|-------------|-----------|------|------|
| 관리자 (본인) | ID/PW | 전체 Read/Write | 시스템 전체 관리 |
| 네팔 유학원 | ID/PW | 학생/결석 Write | 담당 학생만 접근 |
| 대학교 국제처 | 발급된 ID/PW | Read Only | 소속 학생만 조회 |

- 대학교별 고유 ID/PW 발급 및 비밀번호 변경 기능
- 접속 로그 기록 (대학교별)
- Row-level Security (RLS) 적용: university_id 기반 데이터 분리

### 5.2 학생 정보 관리

#### 5.2.1 필수 입력 정보

| 필드명 | 필수 여부 | 비고 |
|--------|-----------|------|
| 대학교명 | 필수 | 드롭다운 선택 |
| 학번 | 필수 | 대학교+학번으로 고유 식별 |
| 학과 | 필수 | - |
| 프로그램 구분 | 필수 | 어학연수/학사/석사/박사 |
| 이름 | 필수 | - |
| 주소 | 필수 | - |
| 휴대폰번호 | 필수 | - |
| 이메일주소 | 선택 | 아르바이트 서류 발송용 |
| 졸업/수료 여부 | 필수 | 재학중/졸업/수료/자퇴/제적 |

#### 5.2.2 데이터 설계 원칙

- 내부 PK: UUID 사용
- 보조 Unique Key: university_id + program + student_no
- 개인정보 동의서: 네팔어 서면 동의 후 스캔 업로드

### 5.3 정기 정보 업데이트 (Quarterly Check-in)

학생 연락처 변경 및 Ghosting(연락두절) 대응을 위해 3개월마다 정기 점검을 수행합니다.

#### 5.3.1 점검 프로세스

| 항목 | 내용 |
|------|------|
| 점검 주기 | 매 분기 (3개월마다) |
| 점검 방식 | 네팔 유학원 대표가 학생과 대면 미팅 시 확인 |
| 점검 항목 | 휴대폰번호, 주소, 이메일 일치 여부 |
| 기록 방식 | 체크박스로 일치 여부 확인 |

#### 5.3.2 시스템 기능

- 분기별 점검 대상 학생 목록 자동 생성
- 연락처 일치 확인 체크박스 (휴대폰/주소/이메일 각각)
- 연락처 변경 시 변경 이력 로그 자동 기록
- 점검 완료일 기록
- 미점검 학생 알림 (점검 기한 초과 시)

#### 5.3.3 변경 이력 로그

| 필드 | 설명 |
|------|------|
| student_id | 학생 고유 ID |
| field_name | 변경된 필드명 (phone/address/email) |
| old_value | 변경 전 값 |
| new_value | 변경 후 값 |
| changed_by | 변경자 (네팔 유학원 대표/관리자) |
| changed_at | 변경 일시 |
| check_in_date | 점검일 |

### 5.4 결석 관리

#### 5.4.1 결석 기록 기능

- 결석 발생 시에만 기록 (출석은 관리하지 않음)
- 기록자: 네팔 유학원 대표, 관리자
- 결석 사유: 질병 / 개인사정 / 기타
- 증빙 서류 업로드 기능 (진단서 등)

#### 5.4.2 증빙 파일 관리

- 파일명: UUID로만 저장 (개인정보 노출 방지)
- 메타데이터(DB): university_id, student_id, absence_date 매핑
- 보관 기간: 졸업/수료 후 1년
- 파기 방식: Soft delete → 일정 기간 후 Hard delete

### 5.5 알림 시스템

#### 5.5.1 결석 알림 (이메일)

결석 발생 시 대학교 국제처 담당자에게 이메일로 실시간 알림을 발송합니다.

| 항목 | 내용 |
|------|------|
| 발송 채널 | 이메일 |
| 수신자 | 국제처 담당자 최대 2명 (최소 1명 필수 등록) |
| 발송 내용 | 학생명, 학번, 결석일, 사유 |
| 발송 시점 | 결석 기록 입력 즉시 |

**이메일 템플릿:**

```
제목: [OO유학원] 학생 결석 알림 - #{학생명} (#{학번})

안녕하세요, OO유학원입니다.

다음 학생의 결석을 알려드립니다.

- 학생명: #{학생명}
- 학번: #{학번}
- 결석일: #{결석일}
- 사유: #{사유}

문의사항이 있으시면 연락 주시기 바랍니다.

감사합니다.
OO유학원 드림
```

#### 5.5.2 담당자 이메일 관리

- 대학교별 담당자 이메일 최소 1명 필수, 최대 2명 등록
- 담당자 변경 시 대학교로부터 서면(이메일)으로 변경 요청 접수
- 이메일 발송 실패 시 로그 기록 및 재발송 처리

#### 5.5.3 민감정보 처리 원칙

결석 알림 시 진단서, 질병명 등 민감 개인정보/건강의료정보는 전송하지 않습니다. 이름, 학번, 결석 사유(질병/개인사정/기타)만 전송합니다.

### 5.6 월간 리포트

대학교 국제처장에게 보고 가능한 형태의 PDF 리포트를 요청 시 생성(On-demand)합니다.

#### 5.6.1 리포트 구성

1. 총 관리 학생 수
2. 해당 월 결석 건수 및 사유별 분류
3. 결석률 (결석 건수 / 총 학생 수 × 수업일수)
4. 3회 이상 결석 학생 명단 (위험군)
5. 전월 대비 변화
6. 비자 갱신 예정자 명단 (해당 시)

#### 5.6.2 리포트 필수 표기 사항

모든 리포트 상단에 다음 문구를 고정 표기합니다:

> **※ 본 리포트는 OO유학원의 학생 관리 활동에 대한 '정보 제공/관리 기록' 목적으로 작성되었습니다.**

#### 5.6.3 리포트 생성 및 제공 방식

| 항목 | 내용 |
|------|------|
| 생성 방식 | 요청 시 생성 (On-demand) |
| 저장 방식 | 생성 즉시 저장하지 않음 |
| 제공 방식 | 짧은 만료 링크로 제공 (24시간 유효) |
| 파일 형식 | PDF |
| 발송 대상 | 대학교 국제처 대표 이메일 |

#### 5.6.4 다운로드 감사 로그

PDF 및 모든 파일 다운로드 시 감사 로그를 기록합니다. 로그에는 다운로드 일시, 다운로드한 사용자, 파일명, IP 주소가 포함됩니다.

### 5.7 아르바이트 서류 지원

외국인 유학생이 합법적으로 아르바이트를 하기 위해 출입국관리소에 제출해야 하는 서류를 제공하고, 학생 정보에 입력된 이메일로 발송합니다. 이를 통해 학생들의 불법 취업 리스크를 사전에 예방합니다.

## 6. 비기능 요구사항

### 6.1 보안

| 항목 | 요구사항 |
|------|----------|
| 데이터 분리 | Row-level Security (RLS) - university_id 기반 |
| 인증 | 대학교별 고유 ID/PW, 비밀번호 변경 기능 |
| 접속 로그 | 대학교별 접속 로그 기록 |
| 다운로드 로그 | PDF/파일 다운로드 감사 로그 기록 |
| 파일 저장 | UUID 파일명, 메타데이터 분리 저장 |

### 6.2 데이터 관리

| 항목 | 요구사항 |
|------|----------|
| 백업 | 일간 자동 백업, 별도 저장소 보관 |
| 증빙 파일 보관 | 졸업/수료 후 1년간 보관 후 자동 파기 |
| 학생 데이터 보관 | 졸업/수료 후 1년간 보관 (동의서 명시) |
| 리포트 PDF | 저장하지 않음, 요청 시 생성 후 만료 링크 제공 |
| 삭제 요청 | 데이터 삭제 요청 처리 프로세스 수립 |

### 6.3 데이터 보관 및 파기 정책

| 데이터 유형 | 보관 기간 | 파기 방식 |
|-------------|-----------|-----------|
| 학생 개인정보 | 졸업/수료 후 1년 | Soft → Hard delete |
| 결석 기록 | 졸업/수료 후 1년 | Soft → Hard delete |
| 증빙 파일 | 졸업/수료 후 1년 | Soft → Hard delete |
| 월간 리포트 PDF | 저장하지 않음 | 요청 시 생성, 24시간 후 링크 만료 |
| 접속/다운로드 로그 | 3년 | 자동 파기 |

### 6.4 법적 준수사항

| 항목 | 요구사항 |
|------|----------|
| 개인정보 동의 | 네팔어 동의서 서면 수집 → 스캔 업로드 |
| 제3자 제공 동의 | 대학교 정보 공유에 대한 명시적 동의 |
| 데이터 소유권 | ERP 내 모든 학생 정보 소유권은 법인에 귀속 |

## 7. 운영 정책

### 7.1 네팔 유학원 계약 조항

| 조항 | 내용 |
|------|------|
| 결석 기록 누락 | 적발 시 페널티 부과 |
| 허위 입력 | 손해배상 책임 |
| 정기 점검 의무 | 분기별 Quarterly Check-in 수행 필수 |
| 데이터 소유권 | ERP 입력 정보는 법인 소유 |

### 7.2 대학교 담당자 관리

결석 알림 이메일 수신자는 대학교당 최소 1명 필수, 최대 2명으로 지정합니다. 담당자 변경 시 대학교로부터 변경될 이메일 주소를 서면(이메일)으로 입력받아 처리합니다.

### 7.3 학생 상태 관리

- 졸업/수료/자퇴/제적 시 해당 상태로 변경
- 상태 변경 후 1년 경과 시 데이터 자동 파기 대상
- 파기 전 대상 목록 관리자 확인 후 처리

## 9. 리스크 및 대응 방안

| 리스크 | 영향도 | 대응 방안 |
|--------|--------|-----------|
| 네팔 유학원 의존도 | 높음 | 계약서 페널티/손해배상 조항 |
| 결석 기록 누락 | 중간 | 월 1회 대면 미팅으로 교차 확인 |
| 학생 연락두절 (Ghosting) | 높음 | Quarterly Check-in으로 조기 감지 |
| 연락처 변경 미반영 | 중간 | 분기별 연락처 일치 확인 |
| 대학교 시스템 접속 저조 | 낮음 | 이메일로 푸시 (접속 불필요) |
| 개인정보 이슈 | 중간 | 서면 동의서 수집, 민감정보 미전송 |

## 10. 부록

### 10.1 용어 정의

| 용어 | 정의 |
|------|------|
| RLS | Row-level Security. DB 수준에서 행 단위 접근 제어 |
| 우수인증대학 | 외국인 유학생 유치 관리 우수 대학 (교육부 선정) |
| 위험군 | 결석 3회 이상으로 비자 연장 리스크 있는 학생 |
| Ghosting | 학생이 연락을 두절하고 잠적하는 현상 |
| Quarterly Check-in | 분기별 학생 연락처 및 상태 정기 점검 |
| On-demand | 요청 시 즉시 생성하는 방식 (사전 저장하지 않음) |

### 10.2 개인정보 동의서 포함 사항

네팔어로 작성된 동의서에 다음 내용을 반드시 포함합니다:

1. 수집 항목: 이름, 학번, 학과, 주소, 연락처, 이메일
2. 수집 목적: 유학 생활 지원 및 리스크 관리
3. 제3자 제공: 소속 대학교 국제처
4. 보관 기간: 졸업/수료 후 1년
5. 삭제 요청 권리 안내

### 10.3 감사 로그 기록 항목

#### 10.3.1 접속 로그

- 로그인 일시
- 사용자 ID
- 사용자 유형 (관리자/네팔유학원/대학교)
- IP 주소
- 접속 결과 (성공/실패)

#### 10.3.2 다운로드 로그

- 다운로드 일시
- 다운로드한 사용자 ID
- 파일 유형 (리포트 PDF/증빙 파일)
- 파일 식별자 (UUID)
- IP 주소

#### 10.3.3 연락처 변경 로그

- 변경 일시
- 학생 ID
- 변경 필드 (휴대폰/주소/이메일)
- 변경 전/후 값
- 변경자
- Quarterly Check-in 점검일
